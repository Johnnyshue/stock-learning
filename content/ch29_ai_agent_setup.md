## Ch29 替爺征的馬要怎麼配——讓 AI Agent 能 AFK 的工作流配置

> [Ch27](#27) 講「要蓋什麼」（量化養成路徑），[Ch28](#28) 是第一份實驗成果。
>
> 這章講**「怎麼把馬配好，讓牠替你出征」**——當你說「爺就 AFK（Away From Keyboard）」走開去睡覺、去看診、去陪小孩，那隻 AI agent 要怎麼設定，才能自己跑、自己驗、跑完叫你、出包不亂搞？
>
> 用木蘭辭的「四市」當骨架：**東市買駿馬**（選模型）、**西市買鞍韉**（人機介面與審計）、**南市買轡頭**（一開頭把方向對好）、**北市買長鞭**（回饋驅動）。
>
> ⚠️ 這章是**工具與工作流配置**，不是投資建議。但它直接服務 Ch27 那條「親手蓋可驗證系統」的路——馬配不好，回測引擎也蓋不出來。

### 🐎 為什麼要用「配一隻馬」來想，而不是「下幾個指令」

散戶用 AI 最常見的失敗，不是模型不夠強，是**把 agent 當許願池**：丟一句「幫我寫個回測程式」，然後盯著看牠跑，跑歪了再罵。這跟「牽一匹沒上鞍的馬，期待牠自己馱你過河」一樣。

能 AFK 的 agent 是**配置出來的**，不是許願出來的。木蘭從軍前跑四個市場買齊裝備，不是儀式——是**一套讓馬可控、可審、可驅動的系統**。四市缺一，你就只能站在旁邊盯著，根本 AFK 不了。

| 市 | 木蘭辭 | 對應 | 缺了會怎樣 |
|---|---|---|---|
| 東市 | 買駿馬 | **選模型** | 馬力不夠，複雜任務原地打轉 |
| 西市 | 買鞍韉 | **人機介面 + 審計** | 你看不到牠在幹嘛、控制不了、出事追不到 |
| 南市 | 買轡頭 | **開頭把方向定對** | 方向歪了，跑越快錯越遠 |
| 北市 | 買長鞭 | **回饋驅動** | 沒人抽牠，牠不知道自己跑錯了 |

### 🐴 東市買駿馬——選模型（SOTA 的兩類選項）

「東市買駿馬」是第一步，也是**最容易過度糾結**的一步。馬有好壞，但 90% 的勝負不在馬（呼應 [Ch27](#27)「他的強 90% 不是 Claude」）。先選一匹夠好的，把力氣花在鞍轡鞭上。

截至 2026 年中，前沿（SOTA, State-of-the-Art）coding agent 的模型基本就兩家：

| 陣營 | 主力（日常）| 旗艦（最難的活）| 定位 |
|---|---|---|---|
| **Anthropic** | Claude Opus 4.8 | Claude Fable 5 | coding / 長程 agentic 的業界基準；Claude Code 原生 |
| **OpenAI** | GPT-5.5 | GPT-5.3-Codex | agentic coding 同級競品；Codex CLI 原生 |

> 參考點：GPT-5.2 Thinking 在 SWE-Bench Pro（真實軟體工程評測）寫下 55.6% 的當時新高；Claude Opus 4.8 是 Anthropic 自評最強的 Opus 級、長程自主執行的業界標竿。兩家咬得很緊，**任何「X 完爆 Y」的說法都該打折**——benchmark 領先三個月就被超車是常態。

#### 怎麼選（給散戶的務實版）

1. **跟著你的 CLI 走。** 用 Claude Code 就配 Claude（Opus 4.8 當主力），用 Codex 就配 GPT。介面與模型同源，工具呼叫、hook、權限模型最順。介面比模型重要——這正是西市的事。
2. **主力用「夠好且便宜」，旗艦留給硬骨頭。** 日常回測、爬蟲、清資料用主力模型就夠（Opus 4.8 約 $5 / $25 每百萬 token 進 / 出）；只有在「整段 repo 重構、跨多檔案的長程自主任務」才升級到旗艦（Fable 5 約 $10 / $50，貴一倍）。**不要為了省錢硬用小模型跑複雜任務**——它原地打轉燒掉的 token 比直接上大模型還貴。
3. **effort 比換模型更值得調。** 現代模型有「思考力度」旋鈕（low/medium/high/xhigh/max）。coding 與 agentic 任務開到 high 或 xhigh，往往比換一個更貴的模型還有效。
4. **不要每天追新版。** 馬夠快就好。你的瓶頸是金融 domain 與工程手感（[Ch27](#27)），不是模型差那 2%。

> 🎯 **醫師視角**：選模型就像選一線用藥——**指引推薦的首選藥 90% 的場合就夠用**，罕見難症才上二線。把精力花在「診斷對不對」（南市）而不是「藥牌子夠不夠潮」。

### 🪮 西市買鞍韉——人機介面與審計（AFK 的命脈）

「西市買鞍韉」是**人要坐的地方**。馬再快，沒有鞍韉你坐不上去、控制不了。這一市是 AFK 的真正命脈：**你不在的時候，怎麼看得到、控制得住、出事追得到。**

#### 1. CLI vs GUI——選 CLI（可審計、可自動化）

| | CLI（Claude Code / Codex / Aider）| GUI（網頁 / IDE 外掛）|
|---|---|---|
| 可審計 | ✅ 每個指令、每次改檔都留痕 | ⚠️ 黑箱，難 diff |
| 可自動化 | ✅ 接 hook、CI、cron | ❌ 綁人手點 |
| 可遠端 | ✅ 配 tmux + tailscale 就能手機操作 | ⚠️ 看平台 |
| 上手 | ⚠️ 要會一點終端機 | ✅ 點點點 |

**要 AFK 就選 CLI。** GUI 適合探索、適合給完全不寫程式的人；但「走開讓牠自己跑」這件事，本質需要一個能被腳本驅動、能留下完整稽核軌跡的介面。

#### 2. Git / GitHub——馬蹄印（讓人類審計的根本）

這是整個西市最重要的一塊。**AI agent 必須在 git 裡幹活：**

- **每個有意義的步驟一個 commit**，訊息寫清楚。你 AFK 回來，`git log` 就是牠這幾小時做了什麼的完整自白。
- **開 feature branch，不要讓 agent 直接推 main。** 牠跑歪了，`git reset` 一鍵回到出征前。
- **用 PR（Pull Request）當審計閘門。** Agent 開 draft PR，你回來在 diff 上逐行看——這比盯著終端機 scroll 有效一萬倍。
- **憑證絕不進 git**（呼應 [Ch27](#27) Phase 0）。Alpaca token、API key 一律 `.env` + `.gitignore`，並用 gitleaks / 內建 secret scan 把關。

> 🐎 木蘭辭「東市買駿馬」之後才「西市買鞍韉」——**先有 git 這套鞍轡，馬才坐得住人**。沒有版本控制的 AI agent = 沒韁繩的烈馬，跑得越快你越害怕。

#### 3. 遠端控制——tailscale + tmux（讓你能 AFK 到任何地方）

AFK 的字面意思是「離開鍵盤」，但你還是想偶爾瞄一眼。經典組合：

- **tmux**：把 agent 跑在一個持久的終端機 session 裡。你關掉 SSH、關掉電腦，agent 照跑；下次連回來 `tmux attach` 就接上原本的畫面。
- **tailscale**：把你的跑機（家裡的桌機 / 一台小 VPS）跟手機、筆電組成一個私有網路（WireGuard 底層），不用開 port、不用搞 VPN 設定。在診間用手機 SSH 進去 `tmux attach`，看一眼 agent 跑到哪，安心了再關掉繼續看診。

這套讓「跑機」與「你人在哪」徹底解耦——馬在馬廄裡替你征戰，你騎不騎在上面都行。

#### 4. Stop 時的提醒——ntfy（AFK 的回叫鈴，特別重要）

這是 AFK 的最後一塊拼圖，也最容易被忽略：**agent 跑完了，怎麼通知你？**

你不可能每五分鐘 attach 看一次——那就不叫 AFK。解法是 **agent 停下來時主動推播到你手機**。

推薦 **ntfy**（讀作 notify）：

- 開源、可自架，HTTP pub-sub 模型，吃資源極小（約 20–50 MB RAM，一台最小的 VPS 就夠）。
- 用法蠢到不行：對一個 topic 發一個 HTTP POST，所有訂閱該 topic 的手機 / 桌面立刻收到。**不用註冊、不用 API key、不用設定。**
- iOS / Android 都有原生 app。

跟 **Claude Code 的 Stop hook** 一接就成：agent 回完一輪、準備等你下一步輸入時，Stop hook 觸發 → 打一個 `curl` 到你的 ntfy topic → 手機叮一聲「回測跑完了，3 個策略全輸大盤，等你看」。

```bash
# Stop hook 範例：agent 停下時推播到手機
curl -d "Agent 收工，等你審 PR" ntfy.sh/我的私密topic名稱
```

> ⚠️ **這一塊不裝，整個 AFK 就是假的。** 沒有回叫鈴，你只能站在馬廄門口等——那叫盯著，不叫離開。

### 🦮 南市買轡頭——一開頭把方向定對（轡頭控制馬頭朝向）

「南市買轡頭」是**控制馬頭朝哪**。這一市決定方向——而**方向錯了，馬跑越快，你離目標越遠**。AFK 之前，這一步要做到滿。

#### 核心原則：開頭十分鐘 > 中間十小時

Agent 的失敗 80% 在「一開始就沒搞懂要做什麼」。你 AFK 之後沒人能糾正方向，所以**出發前必須把 SPEC（規格）釘死**。一套有效的開場流程：

1. **寫 SPEC（規格書）。** 不是「幫我做個回測」，而是「輸入是 `data/` 裡的 2 年日線 JSON，規則是跌 X% 買 Y 股，輸出年化報酬 / 最大回撤 / Sharpe，對照組是買進持有 0050，產出一份 markdown 報告」。**規格越具體，AFK 越安全。**
2. **`grill me`（讓 AI 反問你）。** 把 SPEC 丟給 agent，叫它**反過來拷問你**：「你說的『跌 X%』是從哪天起算？要不要扣手續費？資料有缺值怎麼處理？」——這些它 AFK 時會自己亂猜的洞，現在當面補掉。
3. **brainstorming（先發散再收斂）。** 複雜任務先讓 agent 列 3–4 種做法、各自的權衡，你選一個，再動工。別讓它一頭栽進第一個想到的方案。
4. **`/goal`（把目標釘在 session 上）。** 現代 CLI 能設定一個貫穿整個 run 的目標宣告，讓 agent 長程任務中不會跑題。**清楚的目標 + 一次給足的初始規格 = 最大化自主度**，這正是長程 agentic 模型最吃的東西。
5. **JTBD 框架（Jobs To Be Done——它到底被你「雇來」幹嘛）。** 別問「這工具有什麼功能」，問「我雇這隻 agent 是來完成什麼任務（job）」。你雇回測引擎的 job 不是「跑出漂亮數字」，是「便宜地否證壞策略」（[Ch27](#27) 核心原則、[Ch14](#14) EMH）。Job 定錯，工具蓋得再精美也沒用。
6. **`/plan mode`（問好問滿再動工）。** 這是 AFK 前的最後一道閘。讓 agent 先**只規劃、不動手**，把完整計畫寫出來給你核准（現代 Claude Code 用 PresentPlan 工具：先把計畫寫成檔案、再請你按核准才離開 plan mode）。**你按下核准的那一刻，才是真正可以 AFK 的那一刻。**

> 🎯 **醫師視角**：這整套就是**開醫囑前的問診**。`grill me` = 鑑別診斷的反問，JTBD = 搞清楚「病人真正的主訴是什麼」，plan mode = 開刀前的 time-out 核對。問診草率，後面手術再漂亮都是錯的。AFK 等於把刀交給住院醫師——交之前，計畫一定要核到滿。

### 🐎💨 北市買長鞭——回饋驅動（沒人抽，馬不知道跑錯了）

「北市買長鞭」是**驅動與糾正**。長鞭不是用來虐馬，是**讓馬知道方向對不對、該不該繼續**。AFK 時你不在旁邊喊「停！錯了！」，所以這個回饋必須**自動化、即時、不靠人**。

層層遞進的回饋網（由快到慢、由淺到深）：

| 層 | 工具 | 抓什麼 | 速度 |
|---|---|---|---|
| 1 | **Linter / formatter** | 語法、風格、明顯錯字 | 秒級 |
| 2 | **Build / type check** | 編不過、型別錯 | 秒～分 |
| 3 | **Hook（PreToolUse / PostToolUse / Stop）** | 攔截危險操作、改完自動跑檢查 | 即時 |
| 4 | **單元測試（unit test）** | 邏輯對不對 | 分級 |
| 5 | **CI/CD** | 推上去後的全套驗證 | 分～十分 |
| 6 | **形式化驗證（formal verification）** | 數學上證明關鍵不變量 | 重，選用 |

#### 對 AFK 最關鍵的兩件事

**① Hook 的自我修復迴圈（這是 AFK 的引擎）。**

Claude Code 的 Stop hook 能回傳 `"decision": "block"` 加一個理由，**強制 agent 繼續做、而不是停下來等你**。經典模式：

> Agent 說「我做完了」→ Stop hook 自動跑 `pytest` → 測試掛了 → hook 回 `block`「測試還紅的，去修」→ agent 繼續修 → 再跑 → 綠了才真正停 → ntfy 叫你。

這就是「長鞭」的自動化版本：**馬想偷懶停下時，鞭子自動抽一下「還沒到，繼續」**。沒有這個迴圈，AFK 就是「跑一輪就停在那等你」，等於沒 AFK。

> ⚠️ **但 hook 是雙面刃**：`PreToolUse` hook 也該攔住危險操作（`rm -rf`、推 main、碰 `.env`）。長鞭能驅動，也要能**急停**。

**② CI/CD 當最後的審計閘。**

Agent 開 draft PR 後，CI 自動跑整套（lint + build + test）。你 AFK 回來看 PR，第一眼看的是**那個綠勾還是紅叉**——綠勾代表至少機械層面過關，你只需審邏輯與金融正確性；紅叉代表別看了，先讓 agent 修。**CI 是你不在時，幫你站崗的那個人。**

#### 回到 Ch27 的鐵律：回測本身就是金融層的「長鞭」

技術層的長鞭（lint/test/CI）保證「程式沒寫錯」。但 [Ch27](#27)、[Ch28](#28) 的**回測**，是金融層的長鞭——保證「策略真的有 edge，不是 p-hacking 出來的漂亮歷史」。

兩層長鞭缺一不可：

```
   程式對 ──(lint/test/CI 抽)──> 程式不會 crash ✅
   策略對 ──(回測/樣本外 抽)───> 策略真的會賺（而非過擬合）✅
   兩者都過 ────────────────> 才值得碰 paper trade → 真錢
```

### 🚧 誠實的天花板（裝備買齊了也不是萬靈丹）

> 跟 [Ch27](#27) 一樣，我得把話講清楚，否則就是賣你另一個幻覺。

1. **四市配滿，agent 也不會替你想清楚「該做什麼」。** 它能把你定好的 job 做完、做對，但 job 本身——「這個策略值不值得驗」「這個風險能不能承受」——還是你的判斷。**馬再好，方向是騎士的事。**
2. **AFK 不等於不負責。** Agent 在你的 git、你的帳號下幹活，跑出來的東西**掛你的名**。回來一定要審 PR、審 diff，別綠勾一看就 merge。長鞭再自動，最後拍板的是你。
3. **這套的真正價值，是把「盯著」變成「審計」。** 你不再需要逐字看 agent 打什麼，而是看 commit、看 PR、看 CI 綠勾——把注意力從「過程監工」升級到「結果審查」。這個轉變本身，就是從散戶到「自己的工程團隊 leader」的關鍵一步（[Ch27](#27)）。

### ✅ 下一步

**先配西市與北市的最小可行版**，因為這兩市直接決定你「敢不敢 AFK」：

1. 把 [Ch27](#27) 的回測 repo 放進 git，agent 每步 commit、開 PR。
2. 裝一個 ntfy，接上 Stop hook——讓 agent 收工會叫你。
3. 加一個「測試紅就 block 繼續修」的 Stop hook 迴圈。

做完這三件，你第一次能真正對 agent 說「爺就 AFK」——走開去看診，回來時馬已經替你把回測跑完、PR 開好、CI 綠勾亮著，等你審。
